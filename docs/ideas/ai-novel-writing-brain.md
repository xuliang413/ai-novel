# 精炼一页纸：AI 小说写作外脑

## 问题陈述

**HMW：** 如何让一个 10 年 Java 经验的网文作者，借助 MySQL + Neo4j 双存储和 LangChain4j AI 编排，把「角色/线索/世界观管理 → 章节大纲 → AI 生成正文 → 图谱自动更新」这个写作闭环真正跑通，架构经得起百万字长篇和将来 SaaS 多用户扩展？

## 用户画像

- **我（开发+作者）：** 10 年 Java，写长篇网文（百万字+），三种痛点都有：角色几十个记不住状态、大纲偶尔不够细、不想写重复性文字
- **远期 SaaS 用户：** 同类网文作者，每人用自己的 API Key，项目级数据隔离

## 核心痛点 & 解法

| 痛点 | 解法 |
|------|------|
| 记忆负担（忘了谁在哪、谁恨谁、什么伏笔没收） | Neo4j 知识图谱 → Cypher 精准检索当前上下文 |
| 大纲不够细（不知道下章写什么） | ChapterIntent 模型 → 混合策略（细纲可控 + AI 自主推理） |
| 重复性文字（不想写初稿） | LangChain4j → DeepSeek 流式生成，WebSocket 逐字推送 |

## 推荐方向：A（全量方案）

**MySQL 事实源 + Neo4j 双写**，11 种实体、29 种关系、GraphPatch 审阅闭环、Token 预算截断。

选择理由：
- 长篇网文百万字+ → Neo4j 不是可选项，是刚需
- 10 年 Java 经验 → Neo4j 运维/Cypher 调试/双写一致性处理都在能力范围内
- 市面上 AI 写作工具的共同短板是**没有持久结构化记忆** → 这是差异化壁垒

### 双写一致性策略

- MySQL 是唯一权威事实源
- Neo4j 双写允许暂时不一致，GraphPatch 执行失败时不回滚 MySQL
- 章节状态 `PENDING_GRAPH_UPDATE` 标识 Neo4j 同步待重试
- `graph_change_log` + `inversePatch` 提供修复和回滚能力

## 关键假设 & 验证

- [ ] **假设 1：AI 能稳定抽取图谱变更，准确率 ≥80%** → 写 5 章后人工对比 AI 抽取 vs 实际内容
- [ ] **假设 2：Neo4j Cypher 检索确实比 MySQL JOIN 在复杂多跳查询上显著更优** → 50 章数据量 A/B 对比
- [ ] **假设 3：6000 token 的压缩策略不会在 200 章后严重损害生成质量** → 模拟 200 章规模，生成 5 章人工评估

## MVP 范围（第一里程碑）

**In：**
- 后端数据模型（MySQL 完整 DDL + Neo4j 约束/索引）
- 核心 API：项目 CRUD → 角色/线索/地点/物品 CRUD → 章节大纲 → AI 生成章节 → GraphPatch 审阅 → 图谱更新
- 所有 API 通过自动化测试
- Neo4j GraphPatch 白名单执行器
- Token 预算截断（6000 token 目标，三级截断）
- 保留现有 SmartAdmin 框架壳

**Not Doing（本阶段）：**
- 前端页面改动（保留现有 Vue 壳）
- Wizard 立项引导
- 向量检索（只做图查询）
- 仪表盘/统计
- 自然语言操作（`查灭门案线索进展` 这类）
- 多租户智能路由（先用 SmartAdmin 自带用户隔离）

## Not Doing（永久/远期）
- 章节协作/多人编辑 — 单作者场景优先
- 导出 epub/pdf — 先能写，再导出
- 语音输入 — 非核心
- 移动端 — 桌面优先
