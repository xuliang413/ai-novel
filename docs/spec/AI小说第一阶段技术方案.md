## 〇、文档说明

项目初期所有灵感、创意、用户故事和远期规划，保留在 **[《AI小说原始需求与设计思路》](../AI小说原始需求与设计思路.md)** 中。那篇文档不构成实现约束——它是源泉，不是命令。

## 一、系统概述

### 1.1 这是什么

一个帮网文作者管理角色、线索和世界观，让 AI 基于知识图谱上下文来续写章节的写作工具。

核心闭环：**作者维护设定 → 系统从图谱检索当前上下文 → AI 生成章节 → 系统自动抽取出场角色和线索推进 → 作者确认后写回图谱**。下一章 AI 精确知道「谁在哪、谁恨谁、什么伏笔还没收」。

### 1.2 解决什么问题

| 痛点 | 解法 |
|------|------|
| 长篇写到几百章后，角色几十个、线索几十条，记不住谁在哪、什么伏笔还没收 | Neo4j 知识图谱做持久结构化记忆，写每章前精确检索当前上下文 |
| 大纲偶尔不够细，不知道下章具体写什么 | ChapterIntent 模型支持混合策略——作者可写细纲精确控制，也可只给方向让 AI 推理 |
| 知道要写什么但不想写初稿 | LangChain4j 对接 DeepSeek 流式生成，WebSocket 逐字推送到前端 |

### 1.3 和市面工具的区别

NovelAI、彩云小梦、笔神等工具的共同短板：**没有持久结构化记忆**。它们能续写 2000 字，但写不到第 200 章。

本系统的核心壁垒：Neo4j 图谱 + 线索生命周期 + GraphPatch 人机审阅闭环。每章写完后图谱自动更新，下一章 AI 精确检索当前状态。

### 1.4 给谁用

- **第一阶段：** 作者自己使用（开发即使用，边写小说边完善工具）
- **远期：** SaaS 多用户（每位作者独立项目空间，自备 API Key）

## 二、SaaS 多租户架构

### 2.1 隔离方案

系统基于 SmartAdmin 的用户体系，采用 **用户 → 项目** 两级隔离：

```
系统（单实例）
  ├── 用户 A（私有）
  │    ├── 项目 1：剑道独尊
  │    ├── 项目 2：都市神豪
  │    └── 项目 3：废柴逆袭（草稿）
  │
  ├── 用户 B（私有）
  │    └── 项目 1：魔法纪元
  │
  └── 用户 C（私有）
       └── 项目 1：江湖夜雨
```

| 层级 | 存储 | 隔离方式 |
|------|------|----------|
| 用户认证 | MySQL（SmartAdmin 自带） | Sa-Token + JWT |
| 项目元数据 | MySQL | `t_novel_project.create_user_id`（**所有查询强制过滤**） |
| 图谱数据 | Neo4j | 所有节点和关系带 `projectId` 属性，通过 MySQL 校验归属后访问 |
| 用户 API Key | MySQL（AES 加密） | `t_user_api_key.user_id` |

### 2.2 安全原则

- **API 入口双重校验：** 所有 Neo4j 操作前，必须通过 MySQL 校验「当前用户是否拥有该项目」（`projectId + createUserId` 联合校验），防止通过 URL 猜测 projectId 跨界访问
- **MySQL 查询强制过滤 `create_user_id`：** 所有项目相关查询自动带上当前用户 ID，用户 A 不可看到用户 B 的项目名或任何数据
- **Neo4j 图谱隔离：** 标签只表达实体类型（如 `Character`、`Clue`），项目归属统一通过 `projectId` 属性表达，不使用动态标签

### 2.3 用户 LLM 配置

每位用户配置自己的大模型接入信息，**Key 归属用户，所有项目共享**。

#### 2.3.1 写作模型（对话/生成）

目前支持 **DeepSeek**。用户可配置：
- API Key
- 模型名称（如 `deepseek-chat`）
- 生成参数：temperature、maxTokens、timeout 等

表结构预留扩展——后续加入其他模型（通义千问、OpenAI 等）时，用户只需在前端切换 Provider 即可，不要求改表。

#### 2.3.2 功能性模型（向量 / Rerank）

向量化和重排序是功能性能力，不是所有大模型都提供。目前默认 **通义千问**（`qwen3-embedding` / `qwen3-rerank`），用户可自配 Key、模型名称和参数。

- Embedding：对章节摘要做向量化，用于语义相似检索
- Rerank：对检索结果精排

未配置时不启用向量检索，图查询照常工作。

#### 2.3.3 安全处理

- API Key AES 加密存储
- 前端默认展示 `sk-••••••••`，提供「显示」和「测试连接」
- 未配置 Key 时自动降级为 Mock 模式，不阻塞写作流程

### 2.4 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 后端框架 | Spring Boot 3.x + MyBatis-Plus | SmartAdmin 自带 |
| 鉴权 | Sa-Token + JWT | SmartAdmin 自带，直接复用 |
| 关系数据库 | MySQL | 用户、项目元数据、章节正文、日志 |
| 图数据库 | Neo4j | 知识图谱，存角色/线索/地点/关系 |
| AI 编排 | LangChain4j | Java 版 LangChain |
| 写作 LLM | DeepSeek（用户自配 Key/模型/参数） | 目前仅接 DeepSeek，预留 Provider 扩展 |
| 向量 / Rerank | 通义千问（用户自配 Key/模型/参数） | 默认 `qwen3-embedding` / `qwen3-rerank`，未配置时图查询照常 |
| 前端壳 | SmartAdmin（Vue3 + AntDV） | 登录/鉴权/布局/路由直接复用 |

## 三、第一阶段功能范围

### 3.1 核心功能

| 模块 | 功能 | 说明 |
|------|------|------|
| **项目管理** | 创建/编辑/归档小说项目 | 项目归档后不可见但可恢复，每个项目独立配置 Token 预算 |
| **LLM 配置** | 写作模型（DeepSeek）+ 向量/Rerank（通义千问） | 用户自配 Key、模型名称、参数（temperature/maxTokens/timeout）；表结构预留 Provider 扩展 |
| **API Key 测试** | 配 Key 后一键验证可用性 | 后端发短请求，写一个字即停 |
| **角色** | 创建/编辑/归档 | 基础属性（名称/定位/描述）用户随时可改；动态属性（目标/情绪/战力/状态）仅通过写作流程的 GraphPatch 审阅修改，管理页只读 |
| **地点** | 创建/编辑/归档 | 类型（城市/宗门/秘境等），支持地点层级（CONTAINS 关系，最多 3 层） |
| **线索** | 创建/编辑/归档 | 主线/支线/暗线 + PLOT_THREAD/FORESHADOWING 子类型；揭露程度、当前阶段；生命周期 DORMANT→ACTIVE→RESOLVED |
| **物品** | 创建/编辑/归档 | 类型、状态、数量（可消耗/唯一）；数量扣减仅通过写作流程，默认不勾选 |
| **事件** | 创建/编辑/归档 | 关联发生章节号 |
| **金手指** | 创建/编辑/归档 | 来源、限制、副作用阶段、升级路径 |
| **马甲** | 创建/编辑/归档 | 使用场景、是否已被识破；`revealed` 仅通过写作流程修改 |
| **叙事规则** | 创建/编辑/归档 | 每条规则 = 名称 + 自然语言内容 + 优先级（1~5）。写作时按优先级拼成 System Prompt 注入。不进 Neo4j，纯 MySQL 存，用户随时可改全部字段 |
| **卷** | 创建/编辑/归档 | 卷序号、概要（用户手动写，写作检索注入 Prompt）；章节可通过 `volume_id` 关联卷，支持批量勾选归入，卷管理页可查看已归入章节列表 |
| **角色关系** | 定义/查询/归档 | KNOWS/LOVES/HATES/IS_FAMILY_OF 四种；KNOWS 的 relationType 枚举严格校验，不在白名单的类型拒绝写入 |
| **章节管理** | 已写完章节的列表/详情/编辑 | 正文存取，按章节号排序；可关联所属卷（非必选）；已发布章节编辑时提示可能影响上下文 |
| **章节细纲** | 独立模块，可提前为任意章节号创建/编辑细纲 | 存 JSON 场景节拍或纯文本；写作时自动抽取对应细纲注入 ChapterIntent，可超出当前进度提前规划 |
| **AI 写作闭环** | 完整流程见 §3.2 | 三个确认点：上下文审阅（默认跳过，截断时强制展示）、正文审阅（不可跳过）、GraphPatch 审阅（不可跳过） |
| **流式写作** | WebSocket 逐字推送 | 断线后 AI 继续生成完成，通过 `/recover` 拉回（P1） |
| **无 Key 降级** | 不配 Key 时 Mock 走通全流程 | 正文为占位文本，状态机正常流转 |
| **撤销** | `/undo` 撤销最近一次已执行的图谱变更 | 执行 inversePatch，不删正文和日志，只撤全部操作 |
| **图谱查询** | 项目概览/角色关系网/线索推进历史 | 所有查询入口先校验项目归属 |
| **仪表盘** | 项目字数/章节数/角色数/线索数统计 | 纯 MySQL COUNT |
| **写作日志** | 每章生成后自动记录字数/token/耗时 | 纯写入 |

### 3.2 AI 写作完整流程

```
用户发起写作第 N 章
  │
  ├─ 创建写作会话（chapter_generation_session），记录意图快照和上下文
  │    后续所有状态变更、GraphPatch、inversePatch 都绑定此会话
  │
  ├─ ChapterIntent 组装（三个来源自适应，优先级从高到低）
  │    ① 有细纲（chapter_no=N）→ 抽取细纲场景节拍注入
  │    ② 无细纲 + 用户输入方向 → 以用户输入作为 chapterGoal
  │    ③ 无细纲 + 无输入 → 跳过（AI 自主发挥）
  │
  ├─ 细纲/用户输入解析（混合：字符串碰对 Neo4j 已知实体名 + LLM 解析识别事件/线索/未知实体）
  │    → 匹配到的实体进入候选池（优先级高于自动推断），未创建的实体标记提醒
  │
  ├─ 从 Neo4j 检索上下文（本卷概要/前章/角色/线索/关系/地点）
  │    第 1 章候选池为空时：拉项目所有角色和地点作为初始上下文
  │
  ├─ Token 计数 + 三级截断（项目级配置的目标预算和硬上限）
  │    → 上下文审阅：默认跳过。仅当 Token 截断发生时才弹出给用户确认
  │
  ├─ 组装 Prompt → 调用 DeepSeek 生成
  │
  ├─ 自动质检（字数/视角/POV/新实体）——旁注展示，不阻塞流程
  │
  ├─ 正文保存草稿 → 展示给用户
  │
  ├─ 🔒 正文审阅（不可跳过）
  │    ├─ [✅ 通过并生成变更] → 进入 GraphPatch 抽取
  │    └─ [🔄 重写] → 可选附带修改要求，回到生成
  │
  ├─ AI 抽取 GraphPatch（自动，不可取消）
  │    → 抽取失败时：提示用户，允许「重试抽取」或「跳过图谱更新直接发布」
  │
  ├─ 🔒 GraphPatch 审阅（不可跳过）
  │    ├─ 用户可手动修改任何变更的新值（审阅纠正）
  │    ├─ 用户可按单项拒绝（不勾选）
  │    └─ 高风险操作（角色关系/马甲暴露/物品消耗）默认不勾选
  │
  ├─ [✅ 确认更新] → 白名单执行器写 Neo4j → 章节发布 → 写日志
  │
  └─ 写后校验：Cypher 回溯 DORMANT 伏笔，提醒用户是否启动
```

**细纲与检索的关系：**

细纲告诉 AI「这一章要发生什么」——这是**写作意图**。但细纲不告诉 AI：

- 角色上一章结尾是什么情绪（AI 需要知道李四走进暗室时的心情）
- 线索现在推到什么程度（读者已知多少、还剩什么）
- 上一章结尾发生了什么（衔接不能断裂）
- 细纲未提及但和本章线索相关的其他角色（如驱动「灭门真相」的国师此刻可能也在附近）

所以细纲和检索**互不替代**。细纲精确控制剧情方向，检索提供角色/线索/地点的当前状态和故事连续性。两者合并进入 Prompt，AI 才能在知道「写什么」的同时也知道「此刻是什么状态」。此外，即使有细纲，系统仍会检索**本卷概要**（提供比前章更大尺度的段落方向）和细纲未提及但状态相关的实体（如与目标线索 INTERSECTS 的其他线索）。

### 3.3 实体可修改性

| 实体 | 管理页随时可改 | 仅写作流程改（随剧情自动变化） |
|------|--------------|---------------------------|
| **角色** | name、role_type、description | goal、goal_progress、goal_status、current_emotion、emotion_intensity、secondary_emotion、power_level、current_status |
| **地点** | name、type、summary；层级关系（CONTAINS）也支持管理页手动搭 | — |
| **线索** | name、type、sub_type、description、target_chapter、priority、tone | reveal_level、current_stage、clue_status、summary（每章推进后的进展摘要） |
| **物品** | name、type、summary | quantity、item_status |
| **事件** | name、summary、chapter_occurred（用户可能手动补事件） | — |
| **金手指** | name、type、summary、origin、limitation、evolution | current_stage |
| **马甲** | name、type、alias_context、summary | revealed、revealed_to |
| **叙事规则** | 全部字段 | — |
| **卷** | title、summary（number 不建议改，关联章节范围） | — |
| **所有关系** | — | 角色关系/当前位置/出场记录/持有/参与/线索推进等全部关系 |

核心原则：**会被剧情推进自动改变的属性，只在写作流程的 GraphPatch 审阅时修改**，避免用户手动改值与 AI 推进产生冲突。其他基础属性用户可以随时在管理页调整。

所有实体均提供**列表查询和详情查询**。

### 3.4 归档策略

所有实体删除统一称为「归档」：

- MySQL：`deleted_flag = 1`，不物理删除
- Neo4j：`archived = true`，不 DETACH DELETE
- 归档后用户可见（历史只读），不出现在写作检索和候选中
- 可「取消归档」恢复
- AI 不能自主归档任何实体；归档仅由用户主动触发

|------|------|
| Wizard 立项引导 | P1 |
| 写后复盘 / 方向建议 / 卷自动识别 | P1 |
| `/recover` 恢复中断会话 | P1 |
| 断线重连 | P1 |
| 六个仪表盘完整管理面板 | P1 |

## 四、业务建模

### 4.1 建模概览

**为什么要建模？**

长篇网文到最后的信息量超过人脑容量。你写 500 章不需要记住所有东西——写的时候你只聚焦当下一章。但「第 3 章埋的伏笔在第 297 章引爆」这件事，你的脑子做不到。系统必须替你记住。

建模的本质：**把「写作时需要想的事」提前做成结构化数据，写每章时精确检索，不用作者自己记。**

**建模原则：**

1. **所有东西都是节点，所有联系都是关系。** 李四是节点，暗线「灭门真相」是节点，国师府是节点。李四恨王五是关系，李四在国师府是关系，线索被章节推进是关系。

2. **属性分两类：不变的属于设定，会变的属于状态。** 「剑眉星目、性格坚韧」是设定，写一百章不会改。「当前情绪恐惧、位于国师府」是状态，每章都可能变。设定创建时填好就不动，状态跟随剧情通过写作流程自动更新。

3. **写作流程是状态属性的唯一改动入口。** 状态属性的变化必须通过「写作→AI 抽取→作者审阅→确认写入」这一整圈。作者不能绕过这个闭环直接改状态，但对角色的基础描述、线索的长期规划等设定属性，随时可以在管理页修改。

**实体分层：**

| 层级 | 实体 | 角色 |
|------|------|------|
| **结构层** | 项目、卷、章节、细纲 | 承载故事骨架：项目是最外层容器，卷划分故事大段落，章节是基本写作单元，细纲是可选的章节规划 |
| **内容层** | 角色、地点、物品、事件、线索、金手指、马甲 | 构成故事血肉：角色是动作者，地点提供舞台，物品和马甲是角色的延伸，线索串联叙事张力，事件标记关键节点 |
| **约束层** | 叙事规则 | 设定写作边界（字数、文风、平台红线），不参与内容层的复杂交互 |

**双存储分工：**

- **MySQL 存事实和数据**（章节正文、实体元信息、写作日志、会话记录）——权威事实源
- **Neo4j 存关系和状态**（角色当前位置、情绪、线索进度、人物间恩怨、地点层级）——写作时的上下文检索层

**关键设计决策：**

1. **为什么核心情感用独立关系类型，一般社交用 KNOWS + relationType？** 因为 LOVES 和 HATES 是剧情驱动力——它们会直接决定 AI 写对话和冲突的语气。如果把「爱」「恨」「认识」全塞进一个 KNOWS 关系加类型枚举，检索时很难区分轻重。独立类型天然有高可见性。一般社交关系（朋友、盟友、熟人）不影响剧情走向，所以合并为 KNOWS + relationType 省 Token。

2. **Neo4j 标签只表达实体类型，不拼项目 ID。** 不用 `Character_project_10001` 这种动态标签——标签永远是 `Character`、`Clue`、`Chapter`。项目归属统一通过每个节点和关系的 `projectId` 属性表达。好处：Cypher 模板不用拼字符串、索引稳定、不会因为项目多了产生几千种标签。

下面逐个展开每个实体的设计。

### 4.2 核心实体

#### 4.2.1 项目

项目是整个系统的顶层容器——一个项目就是一本小说。所有其他实体（角色、地点、线索、章节、细纲等）全部挂在项目下。用户可以有多个项目，项目之间完全隔离。

**核心属性：**

| 属性 | 说明 |
|------|------|
| 项目名 | — |
| 类型 | 仙侠/玄幻/都市/历史/科幻/悬疑/武侠/奇幻，影响默认叙事规则和世界观模板 |
| 世界观概述 | 一段自然语言描述，注入每章的写作 Prompt，让 AI 理解这个世界的基本规则（如「末法时代灵气稀薄，修仙者需依赖丹药」） |
| 主角名 | 快捷引用，完整的角色信息在角色管理页 |
| 文风描述 | 约束 AI 写作风格（如「白描克制冷峻，少用形容词」），注入每章 Prompt |
| 平台 | 起点/番茄/纵横……选平台时自动加载对应的叙事规则红线（如敏感词、字数惯例） |
| 目标总字数 | 统计参考，质检不以此为准 |
| 每章目标字数 | 质检时的字数基准（AI 生成的字数与此对比 ±20%） |
| Token 预算 / 硬上限 | 控制每章写作时检索上下文的大小。**项目级配置**——仙侠文 30 个角色 8 条线需要更多上下文，都市文 5 个角色 2 条线可以设低一些。默认 6000/8000 |
| 状态 | 写作中/暂停/归档 |

**关系：**
- 项目在 Neo4j 中通过 `CONTAINS` 关系连接卷（`Project -[:CONTAINS]-> Volume`）

#### 4.2.2 卷

卷把一部百万字小说切成「季」或「篇」。你不分卷也行——但不是给你看的，是给 AI 看的。

**核心属性：**

| 属性 | 说明 |
|------|------|
| 卷序号 | 从 1 开始，决定排列顺序 |
| 卷标题 | — |
| 概要 | 一段自然语言，注入每章 Prompt。写第 67 章时 AI 读到「第二卷：京城风云——李四进入京城，在国师眼皮下调查灭门真相，各方势力浮出水面」，AI 知道这一整段的方向，不会被前几章的支线带偏 |

**章节归属是隐式的：** 章节带一个可选的 `volume_id`，创建时选，事后也能批量调。不强制每章归卷。

**卷之间的切换是叙事分水岭：** 进入新卷 → AI 拿到新卷概要 → 场景、基调、角色目标可能完全不同。

**线索可以挂到卷上：** 线索通过 `BELONGS_TO` 关系关联卷。进入新卷后，旧卷的线索虽然还是 ACTIVE 但不进入 Prompt，提醒作者收束。

**关系：**
- 项目包含卷：`Project -[:CONTAINS]-> Volume`
- 卷包含章节：`Volume -[:CONTAINS]-> Chapter`

#### 4.2.3 章节

章节是写作的基本单元。一章生成后先做草稿，审阅通过后发布，同时更新图谱。

**核心属性：**

| 属性 | 说明 |
|------|------|
| 章节序号 | 全局递增，决定章与章之间的 PREVIOUS 链 |
| 标题 | — |
| 摘要 | 300 字以内，存入 Neo4j Chapter 节点用于检索 |
| 正文 | 全文存在 MySQL，不存 Neo4j |
| POV | 视角人物名，生成时约束 AI 从谁的视角叙事 |
| 所属卷 | 可选，关联到哪个卷 |
| 状态 | DRAFT（草稿）→ PENDING_GRAPH_CONFIRM（等确认图谱）→ PUBLISHED（已发布）；PENDING_GRAPH_UPDATE（图谱写入失败等重试）；INTERRUPTED_DRAFT（生成中断的草稿） |

**章节在 Neo4j 里不存正文，只存摘要、标题和 POV**，检索时 AI 读到的是摘要而非原文。

**关系：**
- 前后相连：`(:Chapter)-[:PREVIOUS]->(:Chapter)` ——写第 N 章时顺着链找到第 N-1 章的摘要
- 卷包含章节：`(:Volume)-[:CONTAINS]->(:Chapter)`
- 记录出场：角色/地点/物品/事件通过 `APPEARS_IN` 关系关联本章
- 线索推进：`(:Chapter)-[:ADVANCES]->(:Clue)` ——本章推动了哪条线索

#### 4.2.4 章节细纲

细纲是提前为某章号写的场景规划，和章节一一对应（靠 chapter_no）。

| 属性 | 说明 |
|------|------|
| 章节号 | 和章节的 chapter_no 对应，可超过当前进度 |
| 场景节拍 | JSON 存场景列表，或纯文本自然语言 |
| 摘要 | 文本描述 |

细纲**只存 MySQL**，不进入 Neo4j。写作时系统先查 MySQL 是否有对应细纲，有则解析注入 ChapterIntent。细纲解析用混合方式（字符串扫已知实体名 + LLM 兜底识别事件/未知实体）。

#### 4.2.5 角色

角色有两大类属性，分属不同修改权限。

**设定属性（创建时填，管理页随时改）：**
- 名字、定位（主角/反派/重要配角/次要角色）、基础描述（外貌、性格等常驻信息）

**状态属性（每章随剧情变化，只通过写作流程 GraphPatch 审阅改）：**

| 属性 | 说明 | 设计考量 |
|------|------|---------|
| 当前目标 | 角色现在想干什么，自然语言 | 一个角色一条 |
| 目标完成度 | 0~1，量化进度 | 数值化方便检索时排序和截断判断 |
| 目标状态 | 进行中/已达成/已放弃/转向 | 和完成度互补——有时完成度没变但方向变了 |
| 当前情绪 | 主导情绪（愤怒/恐惧/坚定/绝望……） | 约束 AI 描写角色的语气和行为 |
| 情绪强度 | 1~5 | 没有强度 AI 容易把「紧张」写成同一个调子，分不出咬牙还是崩溃 |
| 次生情绪 | 表层 + 深层并存（表面坚定实则恐惧） | 单情绪太扁平——真实角色是矛盾体 |
| 战力/境界 | 修仙的筑基期、玄幻的一阶武者 | 自然语言不用枚举，不同世界观体系完全不同 |
| 存活状态 | 活跃/暂离/死亡/失踪/未知 | 死亡角色不进候选池；失踪是悬疑刚需 |

**关系网（以角色为中心，所有关系仅通过写作流程修改）：**

| 关系 | 终点 | 含义 |
|------|------|------|
| KNOWS | 角色 | 一般社交关系，带 relationType（朋友/盟友/竞争对手/熟人/下属/敌人）。**枚举严格校验**——不在白名单内的类型拒绝写入 |
| LOVES | 角色 | 单向爱慕，带状态（单恋/两情相悦/过去式），不阻止一人同时 LOVES 和 HATES 同一人 |
| HATES | 角色 | 单向仇恨，带强度 1~5 |
| IS_FAMILY_OF | 角色 | 亲属/师门，带 familyType（父/母/兄/弟/姐/妹/子/女/堂亲/配偶/师父/弟子） |
| CURRENTLY_AT | 地点 | 当前位置——为什么用关系不用属性？因为地点有层级（暗室→国师府→京城），用关系可以沿 CONTAINS 链向上追溯父级地点 |
| POSSESSES | 物品 | 持有 |
| PARTICIPATES_IN | 事件 | 参与某事件 |
| DRIVES | 线索 | **主动推动**某条线索（如李四调查灭门真相） |
| KNOWS_ABOUT | 线索 | **被动知情**某条线索（如王五知道灭门真相的 50%，但不主动调查）。带 `level`(0~1) 和 `isFalse`(是否持有错误认知)，追踪信息不对称 |
| HAS_CHEAT | 金手指 | 拥有金手指，带 `acquiredInChapter`（在第几章获得） |
| HAS_ALIAS | 马甲 | 拥有马甲身份，带 `createdInChapter`（在第几章首次使用） |
| APPEARS_IN | 章节 | 在哪章出场 |

#### 4.2.6 地点

| 属性 | 说明 |
|------|------|
| 名称 | — |
| 类型 | 城市/村镇/建筑/宗门/荒野/秘境/战场 |
| 描述 | 自然语言描述 |

**核心关系：CONTAINS（地点层级）** —— `(:Location)-[:CONTAINS]->(:Location)`，最多 3 层。京城→国师府→暗室。写作时检索 POV 当前位置 → 沿 CONTAINS 向上追溯父级地点 → AI 知道「暗室在国师府下面、国师府在京城里面」。

其他关系：地点有 `APPEARS_IN`（章节出场），角色通过 `CURRENTLY_AT` 指示在此。

全部属性管理页随时可改，没有动态属性。

#### 4.2.7 物品、事件、金手指、马甲

这四种实体都是角色的延伸——物品是角色持有的东西，事件是角色参与的事，金手指是角色的特殊能力，马甲是角色的隐藏身份。

**物品：**

| 属性 | 说明 |
|------|------|
| 名称、类型、描述 | 设定属性，管理页随时可改 |
| 数量 | 可消耗物品（丹药、灵石）才填；唯一物品（断魂刀）不填。**仅通过写作流程 GraphPatch 扣减**，默认不勾选，下限为 0 |
| 状态 | 完好/损坏/摧毁/遗失。仅通过写作流程修改 |

关系：`POSSESSES`（角色持有）、`APPEARS_IN`（章节出场）、`ITEM_OF`（碎片/部件属于某个主物品，如断魂刀是上古神兵的碎片）

**事件：**

| 属性 | 说明 |
|------|------|
| 名称、描述 | 管理页随时可改 |
| 发生章节 | 用户可手动补（回忆之前发生但没录的），也可通过写作流程自动关联 |

关系：`PARTICIPATES_IN`（角色参与）、`APPEARS_IN`（章节发生）、`TRIGGERS`（事件触发线索，如「宗门大比」触发「宗门内斗」线索）

**金手指：**

| 属性 | 说明 |
|------|------|
| 名称、类型、描述、来源、限制、进化路径 | 设定属性，管理页随时可改 |
| 当前副作用阶段 | **仅通过写作流程修改**，追踪能力代价的渐进演化（如「初期反噬·头痛」→「经脉逆行」→「丹田碎裂」） |

关系：`HAS_CHEAT`（角色拥有）、`BOUND_TO`（绑定于物品，如老爷爷戒指）、`RELATES_TO`（关联线索，如暗线「能力代价」关联金手指「万倍悟性」）

**马甲：**

| 属性 | 说明 |
|------|------|
| 名称、类型、使用场景、描述 | 设定属性，管理页随时可改 |
| 是否已被识破、被谁识破 | **仅通过写作流程修改**（暴露身份是剧情关键节点，不能随便改） |

关系：`HAS_ALIAS`（角色拥有）、`KNOWS_ALIAS`（某人知道马甲背后是谁）、`APPEARS_IN`（马甲身份在章节中出现）

#### 4.2.8 线索

线索是整个系统中最特殊、也最核心的实体。它不是角色、不是地点——它是一**条贯穿多章的叙事线**。

**线索的两种形态：**

| 子类型 | 含义 | 典型例子 |
|--------|------|---------|
| **PLOT_THREAD**（线索） | 持续跨章推进的故事线，读者知道它在走 | 主线「宗门崛起」、暗线「灭门真相」 |
| **FORESHADOWING**（伏笔） | 埋下一个信息点，长期休眠，在未来某章引爆 | 第 3 章「藏经阁角落有一本带血手印的古籍」— 第 89 章才被想起 |

区分它们的关键原因：PLOT_THREAD 占 Token、进 Prompt；FORESHADOWING 在休眠期不占 Token、不进 Prompt，只靠写后校验提醒。

**核心属性：**

| 属性 | 说明 | 设计考量 |
|------|------|---------|
| 名称、完整描述 | 设定属性，管理页随时改 | — |
| 类型 | 主线/支线/暗线 | 主线是全篇核心，支线辅助，暗线是读者暂时不知道的 |
| 子类型 | PLOT_THREAD / FORESHADOWING | 决定了这条线索在检索时的行为——占不占 Token |
| 优先级 | 1~5 | 多条 ACTIVE 线索抢 Prompt 预算时，优先级高的优先 |
| 计划收束章节 | 预估在第几章收束 | 规划参考，不强制执行 |
| 情绪基调 | 悲剧/紧张/温情/热血/神秘/黑暗 | 约束 AI 写相关段落时的语气 |

**动态属性（仅通过写作流程修改）：**

| 属性 | 说明 | 设计考量 |
|------|------|---------|
| 当前进展摘要 | 自然语言，每章更新 | AI 知道这条线最新进展是什么 |
| 揭露程度 | 0~1 | 告诉读者多少了——0.2 是刚刚暗示，0.8 是真相快大白了 |
| 当前阶段 | 自然语言（如「线索碎片」「真相浮现」「证据确凿」） | **和揭露程度正交**——暗线可能揭露程度 0.2（读者几乎不知道）但阶段已是「经脉逆行」（代价已很严重） |
| 状态 | DORMANT→ACTIVE→RESOLVED | 决定了检索行为 |

**线索生命周期：**

```
[创建] → DORMANT ─→ ACTIVE ─→ RESOLVED
              ↑          │          │
              └── 暂停 ──┘          │
                                    │
                收束后保留在图谱中（历史追溯）
```

| 状态 | 写前 Prompt | 写后校验 |
|------|:----------:|:-------:|
| **DORMANT** | ❌ 不进 | ✅ 回溯提醒「第 X 章的伏笔和本章出场的角色有关联，是否启动？」 |
| **ACTIVE** | ✅ 每章检索进展+推进者 | — |
| **RESOLVED** | ❌ 不进 | ❌ 不提醒 |

DORMANT 伏笔可能在第 3 章埋下、第 89 章引爆。写第 45 章时系统通过写后校验发现「角色李四在第 3 章有一条 DORMANT 伏笔」→ 提醒你启动。纯 Cypher 查询，不额外消耗 LLM 调用。

**关系网：**

| 关系 | 终点 | 含义 |
|------|------|------|
| DRIVES | 角色→线索 | 角色主动推动线索（李四调查灭门真相） |
| KNOWS_ABOUT | 角色→线索 | 角色被动知情（王五知道一半），带 level(0~1) 和 isFalse（是否错误认知） |
| INVOLVES | 线索→角色 | 线索牵连某角色 |
| ADVANCES | 章节→线索 | 某章推进了线索，记录推进到什么程度 |
| INTERSECTS | 线索→线索 | 两条线在某个章节交汇（如「灭门真相」和「宗门内斗」在第 15 章交汇），带交汇章节号和描述 |
| BELONGS_TO | 线索→卷 | 线索属于哪一卷，卷结束后不再占 Prompt |
| TRIGGERS | 事件→线索 | 事件触发线索（如「宗门大比」触发「宗门内斗」） |
| RELATES_TO | 线索→金手指 | 线索关联金手指（如暗线「能力代价」关联「万倍悟性」） |

## 五、核心业务流程

### 5.1 写作闭环全景

每写一章，系统走完整圈：

```
管理页设定（一次性）          每章写作闭环（重复）
─────────────────      ─────────────────────────
创建项目               开始写作第 N 章
创建角色/地点/线索      　→ 组装写作意图（细纲/即兴/跳过）
设立关系/规则           　→ 检索上下文（本卷 + 前章 + 候选实体）
                       　→ Token 计数 + 截断
                       　→ 拼 Prompt → AI 生成
                       　→ 正文审阅（通过/重写）
                       　→ 抽取 GraphPatch 候选
                       　→ GraphPatch 审阅（确认/纠正/拒绝）
                       　→ 写入图谱 → 章节发布
                       　→ 写后校验 DORMANT 伏笔
```

左边设定做一次管全局，右边闭环每章跑一遍。图谱随写作逐步「长大」——写第 1 章时里面只有你手动建的几个角色，写到第 50 章时里面已有 50 轮累积的角色状态、线索进度、关系网。

**四个关键确认点：**

| 环节 | 默认行为 | 可否跳过 |
|------|---------|:------:|
| 上下文审阅 | 跳过 | ✅（截断时强制展示） |
| 正文审阅 | 展示 | ❌ |
| GraphPatch 审阅 | 展示 | ❌ |
| 图谱写入 | 自动执行 | — |

### 5.2 写作会话

每次写作创建一个 `chapter_generation_session`，它是整个流程的「黑匣子」——所有中间数据都绑在这条会话上。

**会话记录的内容：**

| 阶段 | 存什么 |
|------|--------|
| 开始 | ChapterIntent JSON、上下文快照 |
| 生成完成 | 使用的 Provider、提示词摘要、正文审阅质检 |
| GraphPatch | 候选 GraphPatch JSON、inversePatch JSON、操作批次 ID |
| 结束 | 结果摘要或失败原因 |

**会话状态流转（即状态机）：**

```
IDLE → GENERATING → CONTENT_REVIEW → PATCH_REVIEW → SUCCESS
                ↓         ↓                ↓
           INTERRUPTED  重写→GENERATING   PENDING_GRAPH_UPDATE → PATCH_REVIEW
                ↓                           ↓
        恢复→GENERATING              CONTENT_REVIEW（放弃变更）
                ↓
              FAILED
        恢复→GENERATING
```

| 状态 | 含义 |
|------|------|
| IDLE | 初始状态，没有活跃的写作会话 |
| GENERATING | AI 正在生成正文 |
| CONTENT_REVIEW | 正文已保存，等作者审阅 |
| PATCH_REVIEW | GraphPatch 已抽取，等作者确认 |
| PENDING_GRAPH_UPDATE | 图谱写入失败，等重试 |
| INTERRUPTED | 用户取消或超时中断，可恢复（P1） |
| FAILED | 生成失败（LLM 异常等），**默认重试 3 次后降级 Mock** |
| SUCCESS | 终态：章节发布、图谱更新完成 |

**单章节互斥：** 同一 `projectId + chapterNo` 同一时间只允许一个非终态会话。

### 5.3 ChapterIntent 组装

ChapterIntent 告诉系统「这一章我要写什么」。三个来源，优先级从高到低：

| 优先级 | 来源 | 触发条件 |
|--------|------|---------|
| ① 细纲 | `t_chapter_outline` 中 `chapter_no = N` 有数据 | 系统检测到细纲存在 |
| ② 即兴输入 | 用户输入了一句方向描述（如「李四潜入国师府，在暗室发现密函」） | 无细纲 + 用户输入了内容 |
| ③ 跳过 | 什么都不给，AI 基于图谱上下文自主发挥 | 无细纲 + 用户无输入 |

**三种模式互不冲突：** 有细纲时系统自动抽取，用户不用额外操作。没细纲但知道方向，一句话告诉系统。没方向就让 AI 基于前章和线索自动推理。同一章每次都从这三个来源自适应。

### 5.4 细纲解析

细纲是自然语言，不是结构化数据。系统需要从中提取「提到了哪些角色、地点、线索，以及有什么事件」。

**混合解析策略：**

```
细纲文本 "李四在国师府暗室里发现密函，推进灭门真相，神秘老者在灵脉入口出现"
  │
  ├─ Step 1：字符串碰对
  │    从 Neo4j 查出当前项目的所有角色名、地点名、线索名
  │    逐个在细纲文本中匹配子串
  │    李四 ✅  国师府 ✅  暗室 ✅  灭门真相 ✅
  │    神秘老者 ❌  灵脉入口 ❌  密函 ❌
  │
  ├─ Step 2：LLM 兜底（仅处理 Step 1 未匹配的部分）
  │    输入：细纲原文 + 已匹配清单
  │    输出 JSON：
  │      { "unmatched_characters": ["神秘老者"],
  │        "unmatched_locations": ["灵脉入口"],
  │        "unmatched_items": ["密函"],
  │        "clues_advanced": ["灭门真相"],
  │        "events": ["在暗室发现密函"] }
  │
  └─ Step 3：汇总展示
        已匹配实体 → 直接进入候选池（优先级高于自动推断）
        未创建实体 → 标记提醒，用户可选 [一键创建] [忽略]
        线索推进 → 自动关联为本章 targetClue
```

**为什么不是纯 LLM？** 纯 LLM 每次都要调 API，100ms 延迟 + 消耗 token。大部分细纲提到的实体已经在 Neo4j 里，字符串碰对零成本、零延迟。只有碰对不上的才走 LLM。

### 5.5 上下文检索

写作第 N 章时，前面 N-1 章的写作已经在 Neo4j 里累积了大量数据和关系。系统不是把全部数据塞进 Prompt，而是**渐进式收窄**。

**第一阶段：粗筛来源（写前自动）**

| 来源 | 拉什么 | 为什么 |
|------|--------|--------|
| 本卷概要 | 查 `(:Volume)-[:CONTAINS]->(:Chapter {number:N})` 获取 Volume 的 summary。**未关联卷时跳过** | 提供比前章更大尺度的段落方向，不因前几章走支线而跑偏 |
| 上一章（N-1） | `(:Chapter {number:N-1}).summary`，以及该章出场过的角色/地点 | 故事有连续性，上章出场的人大概率这章还在 |
| POV 当前位置 | `(:Character)-[:CURRENTLY_AT]->(:Location)` + 沿 `CONTAINS` 向上追溯 2 层父级地点。**POV 角色无 CURRENTLY_AT 时跳过（新角色或第 1 章）** | 主角在哪，这章就在哪发生。层级追溯让 AI 知道暗室→国师府→京城 |
| ACTIVE 线索 | 所有 `(:Clue {status:'ACTIVE'})` + 推动它们的角色（DRIVES）+ 关联的金手指（RELATES_TO） | 正在推进的线，相关角色大概率要出场 |
| 细纲/即兴匹配 | 从混合解析步骤拿到的实体清单 | 用户指定的优先级最高 |
| 候选地点 | 上一章出现过 + POV 所在地父级以上地点 | 场景连续性，最多 2 个 |
| 第 1 章兜底 | 候选池为空时拉项目所有角色和地点 | 避免 AI 面对空白上下文 |

**POV 空缺兜底：** 如果用户未设 POV（第 1 章或遗忘），系统自动用项目主角名作为 POV。主角也未设 → 取第一个 PROTAGONIST 角色。如果连角色都没有 → 不设 POV，让 AI 自由选择视角。

**第二阶段：过滤 + 排序**

| 规则 | 动作 |
|------|------|
| 死亡/失踪角色 | 排除（`status=DEAD` / `MISSING`），死者不进入候选池 |
| 同地点角色 | 只保留近 3 章有出场或有 DRIVES ACTIVE 线索的（排除随机路人） |
| POV 角色 | **永远排在第一位，任何截断层都不能移除** |
| 上章出场角色 | 排 P1，故事连续性最高 |
| 推动 ACTIVE 线索的角色 | 排 P2，剧情相关 |
| 同地点/父级地点的其他人 | 排 P3，场景相关但概率较低 |
| 细纲/即兴匹配的实体 | 所有匹配到的直接提升到 P0 |

**第三阶段：分层展示（按 Token 预算动态分配）**

不固定人数。按优先级顺序往下排，Token 预算用完即停：

- **完整状态卡：** 包含角色当前情绪/目标/战力/位置。排到预算剩余空间不够下一个完整卡时停止
- **简短提示：** 排不下的角色 → 一行名字+位置
- **不进入 Prompt：** 剩余排不下的

**线索同理：** ACTIVE 线索按 priority DESC，取 Top 3 完整进展（当前摘要+揭露程度+当前阶段+推进者），与这些线索 INTERSECTS 的关联线索 → 简短提示。DORMANT 伏笔不进入写前 Prompt，不占 Token，由写后校验单独处理。

**检索结果全量来自 Neo4j Cypher 查询**——角色/线索/地点的当前状态是过去 N-1 轮 GraphPatch 累积的成果。每章发布的正文存在 MySQL，检索时只读摘要。

**语义相似检索（补充）：** 向量检索作为图查询的补充——当用户需要「找和本章风格类似的打斗场景」这类模糊语义匹配时，对章节摘要向量做相似度搜索。日常写作检索以图查询为主（~90%），向量为辅（~10%）。

### 5.6 提示词组装

检索完成后，系统把上下文拼接成发给 AI 的 Prompt。

**System Prompt（系统指令）：**
1. 叙事规则按 priority 降序排列 → 拼接成约束段（红线永远在最前面）
2. 项目世界观概述 + 文风描述 + 每章目标字数

**User Prompt（本章任务）：**
1. 本卷概要（如果本章已关联卷）
2. 前章摘要（第 N-1 章的 300 字摘要）
3. 候选角色（完整状态卡 + 简短提示，按 5.5 策略分配）
4. 角色间关键关系（只保留冲突/驱动剧情的，砍普通熟人关系）
5. 目标线索进展 + 关联线索
6. 金手指 / 马甲信息（仅候选角色的）
7. 地点信息（POV 当前地点 + 父级追溯）
8. 写作指令：「请续写第 N 章，POV：xxx，字数：xxx」

提示词模板用配置文件管理，不硬编码在 Service 代码中，方便迭代调整。

### 5.7 Token 预算与截断

Token 预算和硬上限是**项目级配置**（见 4.2.1 项目），创建项目时自动填入系统默认值（目标 6000 / 硬上限 8000），作者可在项目设置中按需调整。同一作者不同书可设不同值。检索时读取项目当前配置。

中文 Token 近似为字符数——不引入独立分词器。

**三级截断：**

| 级别 | 触发阈值 | 动作 | 省 ~Token |
|------|---------|------|----------|
| Level 1 | 超目标预算 | 砍关联地点 2→1、DORMANT 伏笔不提示 | ~400 |
| Level 2 | 仍超 | 砍普通角色关系、前章摘要 300→100 字、非 POV 角色的金手指/马甲 | ~600 |
| Level 3 | 仍超 | 砍目标线索 3→2、完整状态卡角色数削减 | ~800 |

**P0 保护：** POV 角色永远在完整状态卡中，任何截断层都不能移除。ChapterIntent 中标记 `required=true` 的线索/角色 P0 保护。触发 Level 3 时强制弹出上下文审阅，告知用户被砍掉的内容。

### 5.8 自动质检

AI 生成完成后，系统自动跑 **纯算法检查**（零额外 LLM 调用），结果作为旁注展示，不阻塞流程：

| 检查项 | 规则 | 不通过行为 |
|--------|------|-----------|
| 字数达标 | 目标字数 ±20% | ⚠️ 提示偏多/偏少，用户自行决定 |
| 重复内容 | 正文中连续 3 句（以句号/换行为界）完全相同或高度相似 | ⚠️ AI 偶尔陷入死循环，提示可能重复 |
| 禁忌词命中 | 叙事规则中的禁止词列表扫正文 | ⚠️ 列出命中的词语和位置 |

**不做 LLM 质检：** POV 出场和新实体检测在 GraphPatch 审阅时自然覆盖；前章衔接作者在正文审阅时一眼就能判断，不需要 AI 代劳。

### 5.9 正文审阅

写作流程中**第一个不可跳过的确认点**。用户看到 AI 生成的完整正文 + 质检旁注。

| 操作 | 行为 |
|------|------|
| ✅ 通过并生成变更 | 正文保存草稿，进入 GraphPatch 抽取步骤。**用户可在此编辑正文**——通过后以编辑后的版本为准进入 GraphPatch 抽取 |
| 🔄 重写 | 回到 GENERATING。可选附带修改要求（如「不够燃，重写」），**不给原稿**——避免 AI 模仿上一版。不带要求时用同样的 ChapterIntent + 检索上下文重新生成 |

### 5.10 GraphPatch 抽取与审阅

正文审阅通过后，系统进入图谱变更环节。这是整个写作闭环中**第二个不可跳过的确认点**。

**第一步：AI 抽取候选事实。** 把正文发给 LLM → LLM 识别「谁出场了、谁移动了、哪条线索被推进了、出现了什么新角色/物品/地点」。LLM 只输出事实列表，**不生成 Cypher**。

**第二步：后端生成 GraphPatch 变更单。** 后端从 Neo4j 读出当前状态作为 `before`，结合 AI 的建议作为 `after`，生成标准化的 GraphPatch。

**第三步：展示审阅。** 变更摘要卡片，逐条列出本次所有触发的操作，按类型分组：

| 变更分组 | 展示示例 |
|---------|---------|
| 新增实体 | 🆕 角色「国师」、物品「密函」 |
| 出场记录 | 李四、国师、暗室 本章出场 |
| 角色状态变更 | 李四 情绪：坚定→恐惧 |
| 位置移动 | 李四：国师府前门 → 国师府暗室 |
| 线索推进 | 灭门真相 30%→45%，阶段→真相浮现 |
| 关系变更 | +HATES(李四→国师)、+DRIVES(国师→灭门真相)、+POSSESSES(李四→密函) |

**第四步：作者审阅操作：**
- 可手动修改任何变更的新值（审阅纠正，如把 AI 认为的「坚定」改成「表面坚定实则恐惧」）
- 可按单项拒绝（不勾选）
- 高风险操作默认不勾选，必须手动确认

**GraphPatch 白名单安全机制：**

所有 Neo4j 写入必须通过 5 种受控 Cypher 模板，不允许 LLM 或用户直接写 Cypher：

| 模板 | 用途 | 安全措施 |
|------|------|---------|
| MERGE_NODE | 创建/更新节点 | `$props` 字段白名单校验（拒绝注入 `archived=false` 等非法属性） |
| ARCHIVE_NODE | 归档节点（取代删除） | `SET n.archived = true`，不物理删除，AI 不能触发此操作 |
| MERGE_REL | 创建/更新关系 | 校验两端节点存在 + `projectId` 一致 |
| DELETE_REL | 删除关系 | 校验关系存在，AI 不能触发此操作 |
| UPDATE_NODE_PROPS | 部分更新节点属性 | 字段白名单校验 |

**38 种业务操作映射到 5 种模板。** 例如 `MOVE_CHARACTER` → DELETE_REL(旧 CURRENTLY_AT) + MERGE_REL(新 CURRENTLY_AT)；`ADVANCE_CLUE` → UPDATE_NODE_PROPS(Clue) + MERGE_REL(ADVANCES)。

**操作分级策略（按操作类型定默认行为，不依赖 AI 置信度）：**

| 操作类型 | 默认状态 | 说明 |
|---------|:------:|------|
| 出场记录、新增实体、位置移动 | READY ✅ | 纯事实——正文写了就是写了，默认勾选 |
| 角色状态变更（情绪/目标/战力） | READY ✅ | AI 抽取大概率对，错了作者审阅时一眼能看穿 |
| 线索推进 | READY ✅ | 同上 |
| 角色关系变更（LOVES/HATES/IS_FAMILY_OF）、马甲暴露、物品消耗 | 高风险 ❌ | 最易被 AI 误判——对峙≠恨。默认不勾选，必须作者手动确认 |
| 与当前图谱冲突 | CONFLICT ❌ | AI 给的 before 值和 Neo4j 当前事实不一致，默认不勾选，需作者选解决方案 |
| 非法操作（实体不存在/跨项目/缺必填字段） | BLOCKED 🚫 | 不可勾选，只展示原因 |

**inversePatch：** 每个操作都必须生成对应的逆向操作。`MOVE_CHARACTER` forward 是删旧建新，inverse 是删新建旧。`ADVANCE_CLUE` forward 是更新属性+建关系，inverse 是恢复原值+删关系。创建实体的 inverse 是 `archived=true`（不物理删除）。inversePatch 是撤销功能的基础。

**抽取失败处理：** AI 返回格式不对或解析失败 → 提示用户，允许「重试抽取」。**不允许跳过图谱更新直接发布**——没有出场记录的章节是残缺的，会导致下一章检索不到关键上下文。

### 5.11 图谱写入与章节发布

作者确认 GraphPatch 后，系统执行图谱更新。

**写入顺序：** ① 章节状态置 PUBLISHED + 写 `writing_log`（MySQL）→ ② 白名单执行器写 Neo4j → ③ Neo4j 成功后更新 MySQL 实体表动态属性 + 写 `graph_change_log`（APPLIED）。动态属性在 Neo4j 写入成功后才更新 MySQL，保证两边一致。

**Neo4j 写入失败 → 不拖累 MySQL：** 章节标记为 `PENDING_GRAPH_UPDATE`，会话标记对应状态，允许作者重试。MySQL 中的正文和 change_log 不受影响。

**幂等保护：** 每个 GraphPatch 带 `operation_batch_id`。执行前检查此批次是否已写入 → 已写入则跳过（幂等重放）。恢复中断会话时先查 `graph_change_log` 确认 Neo4j 实际写入状态。

**章节摘要向量化：** 图谱写入成功后，系统调用 Embedding 模型（通义千问）将章节摘要转为向量，存入 Neo4j Chapter 节点的 `embedding` 属性。向量用于后续语义相似章节检索（见 5.5）。未配置向量模型 Key 时跳过此步骤。

### 5.12 写后校验

章节发布后，系统自动回溯图谱状态，**纯 Cypher 查询，不消耗 LLM 调用**。

**① DORMANT 伏笔回溯：**

1. 查出本章出场的所有角色
2. 查这些角色是否关联 DORMANT 伏笔（DRIVES 或 KNOWS_ABOUT）
3. 有 → 提醒作者「角色「李四」有一条 DORMANT 伏笔「血手印古籍」」
4. 每条 DORMANT 线索只提醒一次，提醒后记录 `lastAlertedChapter`，不再重复

**② 线索停滞提醒：**

- 查出所有 ACTIVE 线索，对比其最近一次 ADVANCES 的章节号
- 如果和当前章节号差距超过阈值（默认 10 章）→ 提醒「暗线「灭门真相」已 10 章未推进」

**③ 角色失踪提醒：**

- 查出存活/失踪状态的角色，对比其最近一次 APPEARS_IN 的章节号
- 如果超过阈值（默认 15 章）→ 提醒「张六 15 章未出场」

### 5.13 撤销

`/undo` 撤销最近一次已执行的 GraphPatch。

- 读取 `graph_change_log` 中最近一条 `status=APPLIED` 的记录
- 执行其 `inversePatch`
- 撤销只改图谱，不删除正文和日志
- 撤销后对应 GraphPatch 的中间章节（如果后面又生成了几章）的关系链可能受影响——提示但不阻塞
- 只撤销全部操作，不支持单条撤销

### 5.14 无 Key 降级

当用户未配置任何可用的**写作模型** API Key（DeepSeek Key 为空或无效）时，系统自动切换到 Mock 模式。

| 正常模式 | Mock 模式 |
|---------|----------|
| 调 DeepSeek API | 本地生成占位文本 |
| 正常审阅 + GraphPatch | **同样走完审阅→抽取→Patch→确认流程** |

Mock 模式不跳过任何环节——正文是占位文本，但状态机正常流转，方便调试和前端联调。一旦用户配好 Key，下次写作自动切回正常模式。

**向量/Rerank 模型不可用时：** 不触发整体 Mock 降级。只是不启用向量检索，图查询照常工作。
